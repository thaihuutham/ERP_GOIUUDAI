# Session Log — 2026-04-02 08:20 +07

## Mục tiêu
- Xử lý lỗi Assistant UI: `Không xác định được danh tính để truy cập AI assistant.`

## Root cause
- `AUTH_ENABLED=false` nên `JwtAuthGuard` trước đó `return true` sớm mà không set `AUTH_USER_CONTEXT_KEY`.
- `AssistantAuthzService` yêu cầu identity từ CLS để resolve access => thiếu `userId/role` và ném `deny_missing_identity`.
- Frontend API client cũng chưa gửi dev identity header khi chạy auth-disabled.

## Thay đổi đã thực hiện
- Backend:
  - `apps/api/src/common/auth/jwt-auth.guard.ts`
    - Khi auth disabled: inject dev auth user vào `request.user` + CLS (`AUTH_USER_CONTEXT_KEY`, `TENANT_CONTEXT_KEY`).
    - Nhận header dev:
      - `x-erp-dev-role`
      - `x-erp-dev-user-id`
      - `x-erp-dev-email`
      - `x-erp-dev-employee-id`
      - `x-erp-dev-position-id`
    - Enforce `@Roles` ngay cả trong auth-disabled mode dựa trên dev role header.
- Assistant authz compatibility:
  - `apps/api/src/modules/assistant/assistant-authz.service.ts`
    - Fallback company scope cho synthetic dev manager (`dev_*`) khi thiếu mapping employee/org scope.
- Frontend:
  - `apps/web/lib/api-client.ts`
    - Tự gửi dev identity headers khi `NEXT_PUBLIC_AUTH_ENABLED=false` theo role đang chọn trên UI (`erp_web_role`).
- Tests:
  - `apps/api/test/jwt-auth.guard.test.ts`
    - thêm test inject dev identity khi auth-disabled.
    - thêm test enforce `@Roles` với dev role.
  - `apps/api/test/assistant-authz.service.test.ts`
    - thêm test fallback company scope cho synthetic dev manager.
